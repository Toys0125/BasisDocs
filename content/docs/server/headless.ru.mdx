## Введение

Basis предоставляет сборки безголовых клиентов, которые могут подключаться к серверу Basis без отрисовки полного интерактивного клиента. Эти сборки в первую очередь полезны для нагрузочного тестирования, тестирования соединений и проверки поведения сервера при большем количестве игроков.

Сборки безголового клиента для Linux и Windows также можно загрузить из GitHub Artifacts для Basis. Docker — удобный способ запустить множество экземпляров, но он не обязателен для работы безголовых клиентов.

Используйте безголовые клиенты, если хотите:

- Симулировать нескольких пользователей, присоединяющихся к одному серверу
- Измерять поведение сервера под нагрузкой
- Тестировать доступ с паролем перед приглашением реальных пользователей
- Воспроизводить сетевые проблемы с более контролируемыми клиентами

<Callout type="info" title="О чём эта страница">
Эта страница посвящена запуску самого безголового клиента. Для серверной настройки см. страницы развёртывания, конфигурации и мониторинга.
</Callout>

## Получение безголового клиента

Есть два основных способа запустить безголовые клиенты:

- Загрузить сборку безголового клиента для Linux или Windows из [GitHub Artifacts](https://nightly.link/BasisVR/Basis/workflows/build-docker-server/developer?preview)
- Запустить опубликованные безголовые Docker-образы

Docker полезен, если вам нужны воспроизводимые контейнеризованные конфигурации или нужно быстро масштабировать множество экземпляров. Если вам нужно запустить только один или несколько безголовых клиентов, прямая загрузка сборки для платформы также допустима.

<Callout type="warn">
По умолчанию Windows-версия headless собрана с Mono вместо IL2CPP — это ограничение UnityCI для сборки. Mono потребляет примерно в 1,5-2 раза больше памяти, чем IL2CPP, поэтому используйте Linux-версию, если в пакете не указана версия IL2CPP.
</Callout>

## Перед началом

Убедитесь, что целевой сервер уже доступен.

- Трафик Basis использует `4296/udp`
- Пароль сервера по умолчанию — `default_password`, если вы не изменили его в `config.xml` или через переменные окружения
- Если вы тестируете локальный сервер, используйте адрес локальной машины или LAN вместо публичного имени хоста

В документации сервера также отмечается, что значения `config.xml` могут быть переопределены переменными окружения. Это важно, поскольку тот же принцип используется в примерах безголовых клиентов ниже.

## Настройка Docker Desktop
<Steps>
<Step>
Откройте `Сведения о системе` и проверьте, показывает ли безопасность на основе виртуализации `Выполняется`, и содержит ли раздел `Доступные свойства безопасности` пункт `Поддержка базовой виртуализации`.

Если нет, зайдите в BIOS и включите AMD-Vi или Intel VT-d в зависимости от вашего процессора.

Найдите модель вашей материнской платы из пункта `Продукт базовой платы`, чтобы узнать расположение настройки и клавишу входа в BIOS.
</Step>
<Step>
Перейдите на страницу [Docker для Windows](https://docs.docker.com/desktop/setup/install/windows-install/) и установите Docker Desktop.
</Step>
<Step>
Создайте папку для `docker-compose.yml` и скопируйте пример ниже.
</Step>
<Step>
Запустите командную строку или Powershell в этом каталоге. В проводнике можно открыть командную строку, введя `cmd.exe` в строке URL.
</Step>
<Step>
Сначала загрузите/обновите образы Docker командой `docker compose pull` *Не требуется при использовании стандартного `docker-compose.yml`*
</Step>
<Step>
Введите команду `docker compose up -d` для запуска контейнеров
</Step>
<Step>
Используйте `docker ps` для проверки состояния всех контейнеров и дождитесь, пока все покажут healthy. Это займёт около 30 секунд до минуты.
</Step>
<Step>
Используйте `docker compose scale "basis-headless=#`, чтобы увеличить количество контейнеров. Затем вернитесь к предыдущему шагу и повторяйте, пока не закончатся CPU или RAM.

*Замените `#` числом с шагом не более 5, используйте шаг 1 на слабом или ограниченном по RAM оборудовании*

<Callout type="info">
Использование ресурсов можно также измерить с помощью `docker compose stats`
</Callout>
</Step>
</Steps>

## Конфигурация Docker Compose

При запуске Docker через Docker Desktop движок Docker обычно использует WSL2 под капотом, поэтому Linux-образ как правило является правильным выбором. Если вы специально запускаете Windows-контейнеры *например Windows Server 2016*, используйте Windows-образ из пакета Basis-Headless на GitHub.

<Callout type="warn">
По умолчанию используйте Linux-версию Docker-образа, так как она собрана с IL2CPP. Если вы получаете ошибки манифеста, связанные с Linux, нужно использовать версии с суффиксом `-linux`.
</Callout>

### docker-compose.yml

```yml
services:
  basis-headless:
    image: ghcr.io/basisvr/basis-headless:nightly-linux
    pull_policy: always
    environment:
      Port: 4296
      Password: default_password
      Ip: 'server1.basisvr.org'
      HealthCheckEnabled: "true"
      HealthCheckHost: "127.0.0.1"
      HealthCheckPort: 10666
      HealthPath: "/health"
      AvatarFileLocation: ""
      AvatarPassword: ""
    volumes:
      - Basis:/root/.config/unity3d/Basis Unity/Basis Unity # For Sharing Cached files. If running on Windows you should do it this way.
      #- "${APPDATA}/../LocalLow/Basis Unity/Basis Unity:/root/.config/unity3d/Basis Unity/Basis Unity" # If you want to use your own cache and avatar files. Slower on Windows as its having the overhead from WSL2 *Corruption possible
      #- ./config.xml:/app/HeadlessLinuxServer_Data/config.xml # Optional will be overridden when using environment variables
    healthcheck:
      test: ["CMD", "curl", "-fsS", "http://127.0.0.1:10666/health"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 20s
    deploy:
      resources:
        limits:
          memory: 4096M #Highest I ever saw we need to load a 500MB world file.
        reservations:
          memory: 1024M #Recommend amount that basis will actually need.
    labels:
      autoheal-app: true
  autoheal:
    image: willfarrell/autoheal:latest
    network_mode: none
    restart: always
    environment:
      AUTOHEAL_CONTAINER_LABEL: autoheal-app
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
volumes:
  Basis:
```

## Значение параметров

Приведённые выше примеры используют минимальные значения для подключения:

| Переменная | Назначение |
|----------|---------|
| `Ip` | Имя хоста или IP-адрес сервера Basis |
| `Port` | UDP-порт, на котором слушает сервер, обычно `4296` |
| `Password` | Пароль сервера для подключения |
| `HealthCheckEnabled` | Включить или отключить конечную точку HealthCheck |
| `HealthCheckHost` | IP, на котором слушает конечная точка |
| `HealthCheckPort` | Порт, на котором слушает конечная точка |
| `HealthPath` | Путь для получения информации о состоянии |
| `AvatarFileLocation` | Поддержка комбинированного URL avatar#base64 |
| `AvatarPassword` | Пароль аватара |
| `StrictMemoryCleanupEnabled` | Удалять все загруженные текстуры |

Если вы монтируете `config.xml`, рассматривайте его как файловый резервный конфигурационный источник. Если вы также устанавливаете переменные окружения, согласно документации основного сервера переменные окружения переопределяют значения XML.

## Опциональная симуляция голоса с AudioClips

Безголовые клиенты могут опционально симулировать трафик микрофона, транслируя файл `.wav` или `.opus` в качестве голосового аудио.

Basis проверяет папку `AudioClips` в директории данных безголового игрока при запуске. Если один или несколько файлов присутствуют, безголовый клиент случайно выбирает один файл, зацикливает его, кодирует в Opus и отправляет как обычный голосовой трафик.

Если папка не существует, Basis создаёт её. Если папка существует, но не содержит аудиофайлов, безголовый клиент остаётся беззвучным.

Файлы с частотой дискретизации, отличной от 48 кГц, автоматически ресемплируются в моно `48 кГц`.

### Куда помещать файлы

Для обычной распакованной сборки помещайте файлы в папку данных headless:

- `AudioClips/`

Для Docker смонтируйте папку в директорию `AudioClips` данных игрока.

Пример для Linux headless:

```yml
services:
  basis-headless:
    image: ghcr.io/basisvr/basis-headless:nightly-linux
    volumes:
      - ./AudioClips:/app/HeadlessLinuxServer_Data/AudioClips
...
```


## Запуск в масштабе

Docker Compose может запускать несколько безголовых клиентов из одного определения сервиса:

<h3> `docker compose up --scale "basis-headless=10"`</h3> *--scale когда не задано в docker-compose.yml*

В качестве приблизительного ориентира рекомендуется выделять около `1,5 ГБ` памяти на экземпляр. Например, если у вас доступно около `30 ГБ`, можно планировать примерно `20` экземпляров.

<Callout type="warning" title="Загружайте понемногу">
Не пытайтесь запускать все клиенты одновременно. Это перегрузит RAM вашего компьютера, а в данный момент мы не предлагаем скрипт для удобного запуска экземпляров.

Лучший способ — дождаться, пока состояние всех запущенных контейнеров покажет healthy, прежде чем добавлять ещё.
</Callout>

Это лишь начальная оценка. Реальные ограничения зависят от:

- Доступной оперативной памяти
- Ядер CPU
- `PeerLimit` сервера
- Доступной полосы пропускания между безголовыми клиентами и сервером


## Мониторинг во время тестов

Если конечная точка состояния сервера доступна, вы можете использовать её для подтверждения ожидаемого эффекта тестирования. Документация по мониторингу приводит следующий пример ответа:

```json
{
  "listening": true,
  "visitors": "0",
  "capacity": "1024",
  "sent": "0",
  "recv": "0",
  "currentTime": "2025-05-09T20:19:03.200Z",
  "startTime": "2025-05-09T20:19:00.049Z",
  "version": "6"
}
```

Во время безголового нагрузочного теста наиболее полезны:

- `listening` — для подтверждения, что сервер принимает подключения
- `visitors` — для просмотра количества подключённых клиентов
- `capacity` — для сравнения с настроенным лимитом игроков
- `sent` и `recv` — для наблюдения за ростом трафика
- `version` — для подтверждения ожидаемой версии сервера

<Callout type="warn" title="Соответствие версий важно">
В документации сервера явно указано, что код клиента и сервера должен совпадать по версии сервера. Если ваш безголовый образ и целевой сервер собраны из несовместимых версий, попытки подключения могут завершаться неудачей даже при правильных сетевых настройках.
</Callout>

## Устранение неполадок

Если безголовые клиенты не подключаются, сначала проверьте:

- Адрес сервера в `Ip` доступен из контейнера
- Пароль сервера совпадает
- `4296/udp` открыт и правильно проброшен
- Используется правильный образ для вашей среды выполнения контейнеров
- Сервер действительно слушает и работает при запуске
- Версии клиента и сервера совместимы

Если при масштабировании многих экземпляров наблюдается нестабильность, уменьшите количество экземпляров и определите, является ли ограничивающим фактором память, CPU, полоса пропускания или настроенный `PeerLimit` сервера.

Также рекомендуется обеспечить достаточный объём виртуальной памяти для Unity, чтобы избежать кодов выхода 137 (Out of Memory) в Docker.
